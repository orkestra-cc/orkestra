Operator Onboarding · Orkestra

Users, Tenants & Business Units

A one-page mental model for operators who are new to Orkestra. Read top to bottom.

Users are who. Tenants are where.  Users act. Tenants own. Everything billable, entitled, or invoiced is attached to a tenant — never to a user directly.

1Two tiers of tenancy

Orkestra is a host. Two completely separate populations live inside it.

Tier 1 — Internal operator side

The companies that run Orkestra.
  • Internal users — operators, admins, support staff
  • Internal tenants — Tenant{Kind:internal}
  • Decide which modules are enabled, run the platform, see audit logs
  • Manage their own FatturaPA / SDI invoicing (the operator's books)

Tier 2 — External customer side

The customers who use Orkestra.
  • External users — humans who sign up at app.orkestra.cc
  • External tenants — Tenant{Kind:external}
  • Subscribe to AI / RAG / billing services via Stripe
  • Consume capabilities scoped to their tenant's data

2How a user, a tenant, and the things they own fit together

3Two flavors of external tenant

Within Tier 2, every tenant is one of two shapes. The discriminator is the boolean IsCompany.

B2B / Company tenant

IsCompany = true
  • The legal entity (S.r.l., S.p.A., Ltd, Inc., …)
  • Created eagerly during onboarding
  • Usually has multiple member users (employees, admins)
  • Carries full FatturaPA / SDI billing identity
  • Can have divisions — parent/child tenants
Example. "Acme S.r.l." signs up. Three employees join. Subscriptions, invoices, and Stripe customer all attach to the Acme tenant.

Personal tenant

IsCompany = false  ·  signupChannel = self_serve
  • A single-member account container
  • Created lazily on first tenant-requiring action (subscribe, pay)
  • Name defaults to the user's display name
  • Billing party rendered from the User's first/last name
  • Same data model as a company tenant — just a 1-member case
Example. A freelancer signs up, browses, then clicks Subscribe. EnsureTenantForUser mints their personal tenant on the spot.

4What lives on a tenant

If you find yourself looking up a billable thing by user ID, you are looking in the wrong place. Everything below keys on tenantUUID.

Subscriptions
Recurring AI-services plans (Stripe-backed).
Payments & methods
Transactions, cards, Stripe customer ID.
Entitlements
Capability grants (tenantUUID, capabilityID).
Invoices
FatturaPA / SDI invoices issued or received.
Billing identity
VAT, fiscal code, codice destinatario, PEC.
Members
Which users belong to it, with which role.
Sub-tenants
Child workspaces / divisions, if any.
Data scope
RAG documents, agents, prospects — all tenant-scoped.

5Common shapes you will see in admin

Three real-world layouts, drawn the same way every time.

Solo signup

User: alice@x.com | | member v ┌───────────────┐ │ Personal │ │ tenant │ │ IsCompany=F │ │ members: 1 │ └───────────────┘

One human, one personal tenant, one membership. The tenant exists only because Alice subscribed.

Company with employees

Users: bob, carla, dan \ | / \ | / members v v v ┌───────────────┐ │ Acme S.r.l. │ │ IsCompany=T │ │ members: 3 │ └───────────────┘

Three users belong to one B2B tenant. Subscriptions and invoices roll up to Acme, not to the individuals.

Company with divisions

┌───────────────┐ │ Acme Holding │ parent │ FatturaPA ✓ │ └──────┬────────┘ │ ┌──────┴────────┐ │ Acme Italia │ child │ no FatturaPA │ └───────────────┘

Subscriptions live on the consuming workspace. Invoicing walks up parentTenantUUID until it finds a tenant with billing identity.

6Quick reference

UserA human identity. Auth, profile, MFA. Cannot directly own a subscription or invoice.
TenantAn organizational scope. Owns subscriptions, payments, entitlements, invoices, data.
MembershipThe link between a user and a tenant. Carries the user's role inside that tenant.
Tier 1 — internalTenant{Kind:internal}. The operator's own org. Configures modules, runs the platform.
Tier 2 — externalTenant{Kind:external}. A customer. Either a company (B2B) or a personal account.
IsCompanyDiscriminates B2B (true) from personal (false). Drives billing party rendering and lazy provisioning.
EnsureTenantForUserLazy-provisions a personal tenant on first tenant-requiring action.
DivisionA child external tenant under a parent. Billing walks up parentTenantUUID.
FatturaPAItalian electronic-invoicing identity. Required when IsItalianBillable=true.
Stripe customerExternal-tenant field. Stored on the tenant, never on the user.
